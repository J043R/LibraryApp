# Соответствие критериям курсовой работы

Распределение по ТЗ: Ладыгин Максим реализует клиентскую часть, Бодрова
Александра реализует серверную часть и базу данных.

## Обязательное API пользователей

- `GET /api/users` возвращает список пользователей.
- `GET /api/users/:id` возвращает пользователя по идентификатору.
- `POST /api/users` создает пользователя.
- У пользователя есть `id`, `name`, `email`, `age`.
- `name` и `email` обязательны, `age` необязателен, `id` назначает сервер.
- Данные находятся в файловой SQLite и сохраняются после перезапуска.

## HTML, CSS, Git и LocalStorage

- Интерфейс находится в `frontend/*.html`.
- Оформление и адаптивная верстка находятся в `frontend/css/style.css`.
- JavaScript-клиент выполняет запросы через Fetch API.
- В LocalStorage сохраняются токен, id, имя и роль вошедшего пользователя.
- История Git разбита на последовательные этапы разработки.

## Валидация

- Используются библиотеки `class-validator` и `class-transformer`.
- `ValidationPipe` глобально подключен в `backend/src/main.ts`.
- DTO проверяют обязательные строки, формат email, возраст, роли и длину
  пароля. Некорректный запрос получает HTTP 400.

## Контроллеры и сервисы

- `UsersController` и `UsersService` реализуют API пользователей.
- Отдельные контроллеры и сервисы созданы для авторизации, книг и журнала.
- Контроллеры отвечают за HTTP, сервисы содержат прикладную логику.

## Декораторы и провайдеры

- Используются декораторы NestJS `@Controller`, `@Injectable`, `@Get`,
  `@Post`, `@Put`, `@Delete`, `@Body`, `@Param` и `@Query`.
- Собственный декоратор `@Roles` задает категории доступа.
- `RolesGuard` зарегистрирован как глобальный provider через `APP_GUARD`.
- `DatabaseService` и `TokenService` внедряются через dependency injection.

## HTTP и ошибки

- API демонстрирует GET, POST, PUT и DELETE, body, path- и query-параметры.
- Используются `BadRequestException`, `NotFoundException`,
  `ConflictException`, `UnauthorizedException` и `ForbiddenException`.
- Глобальный `ApplicationExceptionFilter` формирует единый ответ для
  непредвиденных серверных ошибок.

## Категории доступа

- Администратор управляет пользователями, книгами и журналом.
- Библиотекарь просматривает пользователей и управляет выдачей книг.
- Читатель просматривает только свой профиль и работает со своими выдачами.
- Ограничения проверяются на сервере guard-ом, а frontend скрывает
  недоступные разделы и команды.

## Дополнительные возможности NestJS

- Модульная структура и dependency injection.
- Глобальные validation pipe, guard и exception filter.
- Собственный параметрический декоратор ролей.
- HMAC-токены с ограниченным сроком действия.
- Раздача статического frontend тем же NestJS-приложением.
- Журналирование операций и предметная логика выдачи/возврата книг.

## SQLite

- `DatabaseService` открывает `backend/data/library.sqlite`.
- Схема создается автоматически при первом запуске.
- Используются четыре связанные таблицы: `users`, `books`,
  `borrowed_books`, `logs`.
- Внешние ключи защищают связи выдач с книгами и читателями.
- Выдача и возврат выполняются внутри SQL-транзакций.
- Все пользовательские значения передаются параметрами, а не вставляются
  в SQL-строки.
